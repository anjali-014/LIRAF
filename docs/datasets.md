# Datasets

This document describes every dataset the Terrain Analysis pipeline uses, where to obtain it, its projection/resolution, and — critically — why the raw datasets are not committed to this repository.

## Why Raw Datasets Are Not Committed

The primary input, `faustini_test_dem_100m.tif`, is approximately **2.5 GB**, far beyond what's appropriate to store in a Git repository. Several intermediate arrays generated by the pipeline are also large (hundreds of MB) and fully regenerable by re-running the notebook. Committing any of these would bloat the repository, slow down every clone, and go against standard practice for scientific-software repositories, which typically document data provenance and let users fetch data independently. See the repository's `.gitignore` for the exact exclusion list.

## Primary Input Dataset

### Faustini DEM (`faustini_test_dem_100m.tif`)

| Property | Value |
|---|---|
| Purpose | Elevation base for horizon-angle modeling, slope, hillshade, illumination, PSR mapping, and landing-site selection |
| Format | GeoTIFF, single band, float32 |
| Resolution | 100 m/pixel |
| Source dataset | Derived from the Lunar Reconnaissance Orbiter (LRO) Lunar Orbiter Laser Altimeter (LOLA) Digital Elevation Model, per the team's BAH 2026 proposal |
| Coverage | Faustini crater region, lunar south pole (~87.3°S) |
| License | LOLA data is produced by NASA/GSFC and distributed via the Planetary Data System (PDS); it is public domain / open-access under standard NASA PDS data policy |

**Where to obtain it:** the LOLA DEM archive is hosted on the NASA Planetary Data System (PDS) Geosciences Node. Search the PDS Geosciences Node or the LOLA PDS4 archive for south-polar DEM tiles/mosaics covering Faustini crater, and derive or crop a 100 m/pixel product covering the same extent as this pipeline's DEM. Exact tile identifiers were not recorded in the source notebook; if you have the specific PDS product ID or download URL used to produce `faustini_test_dem_100m.tif`, add it here for full reproducibility.

**To regenerate everything downstream once you have the DEM:** place it at the path referenced in `config/faustini_params.yaml` (or pass your own path to any `src/terrain_analysis` function) and follow `examples/quickstart.md`.

## Derived / Intermediate Datasets (Regenerable, Not Committed)

| File | Size (approx.) | Regenerate via |
|---|---|---|
| `horizon_angles.npy` | Hundreds of MB (72 × rows × cols, float32) | `horizon_angles.compute_horizon_angles()` — approx. 40–60 min on Colab's free tier per the original notebook's own runtime estimate |
| `PSR_outputs/horizon_chunks/` | Variable, many small files | Automatically created and cleaned up as a resumability mechanism during horizon-angle computation; not a deliverable in itself |
| `illumination_fraction.npy`, `psr_mask.npy` | 1.5–6 MB each | `illumination.compute_illumination_and_psr()` |
| `candidate_ice_traps.npy` | A few MB | `ice_trap_detection.detect_candidate_ice_traps()` |
| `landing_sites.npy` | A few MB | `landing_site_scoring.rank_landing_sites()` |
| Full flattened CSV (`combined_lunar_dataset.csv`, ~1.6M rows) | Tens of MB | `terrain_processing.export_combined_dataset()` |
| `hillshade.tif` | A few MB (small enough to commit, but currently excluded — see note below) | `terrain_processing.save_hillshade()` |

**Note on `hillshade.tif`:** this file is small enough to commit but has been intentionally excluded from this repository's current state per prior repository-management decisions; regenerate it locally if needed.

**Note on the full flattened CSV:** rather than committing the full ~1.6-million-row `combined_lunar_dataset.csv`, this repository includes only `outputs/tables/sample_terrain_data.csv` — a small representative sample showing the schema (`X_m`, `Y_m`, `Elevation_m`, `Slope_deg`, `Illumination_fraction`). Regenerate the full file locally via `terrain_processing.export_combined_dataset()` if you need the complete pixel-grid table.

## Supplementary Datasets Referenced in the Proposal (Not Used by This Repository's Code)

These appear in the team's LIRAF proposal as inputs to *other* modules (radar processing, data fusion) and are documented here only for context — no code in this repository reads them:

| Dataset | Role (per proposal) | Owning module |
|---|---|---|
| Chandrayaan-2 DFSAR | CPR/DOP computation | Radar Processing (Aditya Narayan) |
| Chandrayaan-2 OHRC | Boulder detection, crater morphology, landing safety | Radar/Surface Analysis |
| LRO Mini-RF | Proxy dataset for pre-finale CPR/DOP pipeline validation | Radar Processing |
| Lunar Prospector | Hydrogen abundance | Data Fusion |
| SPICE kernels | Referenced in the proposal for illumination/ephemeris; this repository's actual implementation uses Astropy's DE421 ephemeris plus a manual IAU rotation instead of SPICE kernel files directly — see `docs/methodology.md`, Section 2 |

## Coordinate Reference System

The DEM's exact CRS and spatial bounds can be printed directly from the file using `terrain_processing.print_dem_reference_info(dem_path)`, reproduced from the original notebook's own CRS-inspection cell. Run this after obtaining the DEM to confirm its projection before cross-referencing with other datasets (e.g. in QGIS).
